## Findings — Introduced / Aggravated

### Critical

#### C1. `reOCRCurrentPage` builds `LearnerPageContext` with non-composite `mangaId`, breaking PageKey lookup [Introduced]

**File:** `iOS/UI/Reader/ReaderViewController.swift:600-611`
**Provenance:** `git blame -L 600,615 iOS/UI/Reader/ReaderViewController.swift` → `8635bceab` (this PR's Task 6 commit).

**Trace:**

1. `ReaderPagedViewController.setPage(targetVC:)` (line 444-449) sets `pageView.learnerContext.mangaId = "\(sourceId).\(viewModel.manga.key)"` — **composite** form.
2. `notifyLearnerOfImage` calls `imageDidLoad(image, context: ctx, …)` → coordinator stores `pageStates[PageKey(context)]` keyed by **composite** mangaId.
3. User taps the new "Re-OCR" bar button → `ReaderViewController.reOCRCurrentPage()` (line 600) builds a fresh `LearnerPageContext(sourceId: manga.sourceKey, mangaId: manga.key, …)` — **just `manga.key`**, no source prefix.
4. `LearnerOverlayCoordinator.reOCR(for: context, container: container)` (`LearnerOverlayCoordinator.swift:139-150`):
   - `pageStates[PageKey(context)]?.lastOCRResult = nil` — **lookup misses** because the existing entry was keyed by composite mangaId; this is a silent no-op.
   - `imageDidLoad(image, context: context, container: container)` is called with the **non-composite** context.
5. Inside `imageDidLoad`:
   - `isLearnerEnabled(for: context.mangaId)` reads `Learner.mode.{manga.key}` — that key was never set (the settings sheet writes `Learner.mode.{sourceKey}.{mangaKey}`). Returns default `.inherit` → global state.
   - `existingOverlay = pageStates[PageKey(context)]?.overlay` again misses, so a **new overlay is constructed and added as a subview** to the already-overlaid `container.imageView`. Two stacked `LearnerOverlayView` instances result.

**Triggering scenario:** Open a paged manga with Learner ON, tap the Re-OCR bar button. Expected: cache invalidated, single overlay rebuilt. Actual: cache invalidate runs against the wrong cache key; original overlay remains; a second overlay is added (duplicate badges, doubled tap-targets, doubled OCR work).

**Evidence commands:**
- `grep -rn "LearnerPageContext(" Shared iOS --include="*.swift"`
- `git blame -L 600,615 iOS/UI/Reader/ReaderViewController.swift`
- `grep -n "manga.sourceKey\|manga.key" iOS/UI/Reader/ReaderViewController.swift`

**Fix:** change line 603 to `mangaId: "\(manga.sourceKey).\(manga.key)"` (or `manga.identifier.description`) to match `ReaderPagedViewController.setPage(targetVC:)`.

**Acceptance Criteria impact:** Task 6 #3 ("Tapping it clears the page's OCR cache and re-runs OCR") and #4 ("After a re-OCR, the on-page overlay updates without leaving the page") are **Not Met**.

### Improvements

#### I1. Nested `Section` inside another `Section` in `ReaderSettingsView` [Introduced]

**File:** `iOS/New/Views/Reader/ReaderSettingsView.swift:418-435`
**Provenance:** commit `482402028` (Task 7).

The OCR-languages picker is wrapped in a `Section { … } header: { … } footer: { … }` block that lives inside the outer `Section { … } header: Text("LEARNER_MODE") footer: …` block. SwiftUI's `Form` does not support `Section`-inside-`Section`; the inner section's header/footer typically render as inline rows or are silently dropped, depending on iOS version. Either flatten the inner section into the outer one (move `LearnerOCRLanguagesPicker` directly after the mode `SettingView`) or extract the OCR-languages controls into a sibling `Section` at the same nesting level as the outer one.

#### I2. `LearnerOCRLanguagesPicker` stores selection as `Set<String>` then sorts alphabetically on save — loses user-intended priority order [Introduced]

**File:** `iOS/UI/Learner/LearnerOCRLanguagesPicker.swift:71-77`
**Provenance:** uncommitted (file untracked; committed predecessor was at `iOS/New/Views/Reader/`).

Vision's `VNRecognizeTextRequest.recognitionLanguages` uses array order as a priority hint. Storing as `Set` and emitting `sorted()` means German always precedes Japanese alphabetically (`"de-DE" < "ja-JP"`), regardless of what the user actually emphasises. Consider keeping an ordered representation (`[String]` with insertion order, or a `Set` for membership + a sibling `[String]` for order).

#### I3. `ReaderViewController` Re-OCR bar button visibility is evaluated once at view-controller setup, not observed [Introduced]

**File:** `iOS/UI/Reader/ReaderViewController.swift:166-177`
**Provenance:** commit `8635bceab` (Task 6).

`navigationItem.rightBarButtonItems` is computed inside the navbar-setup block, with `LearnerGate.isEnabled(mangaId: mangaCompositeId)` read once. If the user opens a manga with Learner off, toggles per-manga to On in the reader settings sheet, the Re-OCR button will not appear until they leave and re-enter the reader. Either re-evaluate after the `Learner.mode.{compositeId}` notification fires, or always include the button and hide it via `isHidden`/`enabled` on the gate.

#### I4. `WordLookupViewModel.editableEntry` is `weak`, leading to silent no-op if the entry faults out [Introduced]

**File:** `iOS/UI/Learner/WordLookupViewModel.swift:40-41`
**Provenance:** commit `8635bceab` (Task 5).

`editableEntry` holds the CoreData object weakly. If the list that opened the sheet refreshes its fetch (releasing the strong ref) while the sheet is up, the entry could deallocate; `applyEdits` then returns silently (`guard let entry = editableEntry else { return }`). Consider holding the entry's `NSManagedObjectID` instead and re-resolving at save time — gives the same memory profile but immune to fault eviction.

#### I5. `ReaderGestureDelegateTests` duplicates production predicate logic in the test [Introduced]

**File:** `AidokuTests/ReaderGestureDelegateTests.swift:25-32`
**Provenance:** commit `1ca92feb` (Task 2).

The test defines `shouldReceiveTouch(on:)` locally and asserts against it, rather than against the actual `gestureRecognizer(_:shouldReceive:)` on `ReaderViewController`. A bug in the production predicate (e.g., the recognizer-identity guard at line 1262) would not be caught — the local helper omits that guard entirely. Refactor the production predicate into a free function or static helper that both the production code and the test call, so the test exercises the actual code path.

#### I6. `SentenceTranslationSheet.task(id: viewModel.sentences.count)` may re-fire and re-highlight as translations stream in [Introduced]

**File:** `iOS/UI/Learner/SentenceTranslationSheet.swift:133`
**Provenance:** commit `8635bceab` (Task 6).

Using the sentence count as the task identity means the highlight animation re-runs whenever the list grows (initial seed → translated → simplified). Likely a single re-trigger in practice, but the user could see the highlight reappear after they've already scrolled away. Consider gating on `viewModel.sentences.isEmpty == false` and tracking "already highlighted once" in a `@State Bool`.

#### I7. `LearnerOCRLanguagesPicker.saveToDefaults` always writes even when the toggle change was prevented by the "last-language" guard [Introduced]

**File:** `iOS/UI/Learner/LearnerOCRLanguagesPicker.swift:40-50`
**Provenance:** uncommitted.

When `selected.count == 1 && newValue == false`, the `else if selected.count > 1` branch is skipped — `selected` is unchanged — but `Self.saveToDefaults(selected)` still fires (writing the same value back). Minor; moves the save inside the two mutating branches.

#### I8. OCR-languages migration only runs when `LearnerOverlayCoordinator.ocrLanguages()` is called [Introduced]

**File:** `Shared/Learner/Reader/LearnerOverlayCoordinator.swift:188-208`
**Provenance:** commit `48240202` (Task 7).

The migration from `Learner.ocrLanguages` (String) → `Learner.ocrLanguagesList` ([String] JSON) fires lazily inside `ocrLanguages()`. `LearnerOCRLanguagesPicker.loadFromDefaults()` reads `Learner.ocrLanguagesList` directly; if the user opens Settings before opening any reader page, the picker shows the default `["de-DE"]` and overwrites the user's legacy single-language preference. Either run the migration eagerly at app launch, or have the picker call into `LearnerOverlayCoordinator.shared.ocrLanguages()` once on `init`.
