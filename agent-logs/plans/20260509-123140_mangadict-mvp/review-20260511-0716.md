## Findings (Introduced / Aggravated)

### Critical

**C1 — `0.9.0.xcdatamodel` is on disk but not registered in `XCVersionGroup`; the built `.momd` will keep using 0.8.2. [Introduced]**

`Aidoku.xcodeproj/project.pbxproj:3766-3778`:
```
FB1B482B277CE18D0056D02C /* Aidoku.xcdatamodeld */ = {
    isa = XCVersionGroup;
    children = (
        FB0AB8922F50B81800AF1451 /* 0.8.2.xcdatamodel */,
        ...                  (8 older versions, no 0.9.0)
    );
    currentVersion = FB0AB8922F50B81800AF1451 /* 0.8.2.xcdatamodel */;
    ...
};
```

`Shared/Aidoku.xcdatamodeld/.xccurrentversion` says `_XCCurrentVersionName = 0.9.0.xcdatamodel`, but the pbxproj `XCVersionGroup` has no `0.9.0.xcdatamodel` child and `currentVersion` still references the 0.8.2 file. Xcode uses the `XCVersionGroup` children list to decide which `*.xcdatamodel` directories to compile into the bundled `.momd`. Since 0.9.0 is not in the children list, it will not be compiled, and `NSPersistentContainer(name: "Aidoku")` at runtime will use the 0.8.2 model — which lacks `VocabularyEntry`/`FamiliarityProgress`/`FlashcardState`.

Trigger trace (production runtime):
1. App launches; `CoreDataManager.shared.container` is realized.
2. `NSPersistentCloudKitContainer(name: "Aidoku")` loads `Aidoku.momd`.
3. `.momd` contains only versions 0.5 → 0.8.2 — entity `VocabularyEntry` is undefined.
4. Any call to `VocabularyEntryObject(context: ctx)` or `getVocabularyEntry(...)` will crash with `NSEntityDescription not found for VocabularyEntry`.

Provenance: `git diff Aidoku.xcodeproj/project.pbxproj` shows added build-file entries for the Swift files but no `XCVersionGroup` child addition. `git blame` is unnecessary — the changes are uncommitted and the missing-line analysis is in the diff itself.

Fix: in pbxproj, add `0.9.0.xcdatamodel` to the `XCVersionGroup` children, set `currentVersion` to it, and add a `PBXFileReference` for the new model. Easiest path: open the project in Xcode, select the `Aidoku.xcdatamodeld` group, and let Xcode insert the new model version through the UI (Editor → Add Model Version) — this also handles `.xccurrentversion` and the `currentVersion` field consistently.

**C2 — Duplicate codegen: entities declare `codeGenerationType="class"` AND the hand-written subclass files redeclare the class + `@NSManaged` properties → compile error. [Introduced]**

`Shared/Aidoku.xcdatamodeld/0.9.0.xcdatamodel/contents:113,132,139`:
```
<entity name="VocabularyEntry"      ... codeGenerationType="class">
<entity name="FamiliarityProgress"  ... codeGenerationType="class">
<entity name="FlashcardState"       ... codeGenerationType="class">
```

`Shared/Data/Database/Objects/VocabularyEntryObject.swift:12-13, 51-66`:
```swift
@objc(VocabularyEntryObject)
public class VocabularyEntryObject: NSManagedObject { ... }
extension VocabularyEntryObject {
    @nonobjc public class func fetchRequest() -> NSFetchRequest<VocabularyEntryObject> { ... }
    @NSManaged public var id: UUID
    @NSManaged public var lemma: String
    ...
}
```

Same shape in `FamiliarityProgressObject.swift:12-31` and `FlashcardStateObject.swift:13-27`.

With `codeGenerationType="class"`, Xcode generates `VocabularyEntry+CoreDataClass.swift` (containing the class) and `VocabularyEntry+CoreDataProperties.swift` (containing the `@NSManaged` extension) in DerivedData at build time. Combined with the hand-written file, the build will see:
- two `public class VocabularyEntryObject: NSManagedObject` declarations → "invalid redeclaration of 'VocabularyEntryObject'"
- two `@NSManaged public var lemma: String` (and every other field) → "invalid redeclaration of 'lemma'"
- two `fetchRequest()` declarations → "invalid redeclaration of 'fetchRequest()'"

The existing project convention disambiguates these patterns clearly:
- Entities with **no `codeGenerationType` attribute** (Manga, Chapter, History, LibraryManga) → manual codegen; hand-written file declares class + `@NSManaged`. See `MangaObject.swift:13` and `:171-196`.
- Entities with **`codeGenerationType="class"`** (Track, Source, MangaUpdate, Category, etc.) → Xcode generates everything; hand-written file is extension-only with helpers. See `TrackObject.swift` (just `extension TrackObject { func toItem() ... }`).

The plan's Decision #2 ("`codeGenerationType="class"`") and Decision #3 ("matches `MangaObject.swift`") are contradictory — the executor picked both, producing this conflict.

Inconclusive note: I could not run `xcodebuild` in this review environment (Command Line Tools active, not full Xcode). The conclusion above is from reading the model XML, the hand-written files, and comparing to the established convention — all three of which agree. Resolve by: (a) build with `xcodebuild -project Aidoku.xcodeproj -scheme Aidoku build`, expect duplicate-symbol errors; (b) fix by **either** removing `codeGenerationType="class"` from each of the three entities (mirroring MangaObject's manual-codegen pattern, keeping all hand-written declarations), **or** stripping the class declaration and `@NSManaged` block from each hand-written `*Object.swift`, leaving only the extension helpers (mirroring TrackObject's class-codegen pattern). Either is fine; pick the one that matches the maintainer's preference (TrackObject's pattern is more common in this model).

**C3 — Test `setFamiliarity_capsAtThree` asserts `correctAnswers == 3` after four correct reviews; the implementation will produce `correctAnswers == 4`. [Introduced]**

`AidokuTests/VocabularyManagerTests.swift:69-84`:
```swift
@Test func setFamiliarity_capsAtThree() throws {
    ...
    for _ in 0..<4 {
        CoreDataManager.shared.markFlashcardReview(entry, correct: true, context: ctx)
    }
    let refetched = CoreDataManager.shared.getVocabularyEntry(language: "de-DE", lemma: "wasser", context: ctx)
    #expect(refetched?.progress?.level == 3)
    #expect((refetched?.progress?.correctAnswers ?? 0) == 3) // 4 calls but level locked at 3 after 3
}
```

`Shared/Managers/CoreData/CoreDataManager+Vocabulary.swift:131-147`:
```swift
func markFlashcardReview(_ entry: ..., correct: Bool, ...) {
    ...
    if correct {
        progress.correctAnswers += 1                        // always increments
        if !progress.done && progress.level < 3 {
            progress.level += 1
        }
    }
    progress.lastReviewedAt = Date()
    try? ctx.save()
}
```

Trigger trace:
1. Test calls `markFlashcardReview(entry, correct: true, ctx)` 4 times.
2. Each call hits the `if correct` branch and runs `progress.correctAnswers += 1` unconditionally.
3. After 4 calls, `correctAnswers == 4`.
4. The assertion `#expect((refetched?.progress?.correctAnswers ?? 0) == 3)` fails.

The plan's documented rule (Approach section, last paragraph) is "always increment `progress.correctAnswers` if correct". The implementation matches that rule. The test's inline comment ("4 calls but level locked at 3 after 3") shows the author was thinking about `level` (which is correctly asserted to be 3 on the line above) and accidentally re-used the same number for `correctAnswers`.

Fix: change the assertion to `== 4`, or split the test into two `#expect`s with clearer intent — `level == 3` (capped), `correctAnswers == 4` (always increments).

Provenance: file is new and untracked — `git status` confirms `?? AidokuTests/VocabularyManagerTests.swift`.

### Improvements

**I1 — `VocabularyEntryObject.normalize(_:)` trims `.whitespacesAndPunctuation` but is never called. [Introduced]**

`Shared/Data/Database/Objects/VocabularyEntryObject.swift:26-28` defines:
```swift
static func normalize(_ surfaceForm: String) -> String {
    surfaceForm.lowercased().trimmingCharacters(in: .whitespacesAndPunctuation)
}
```

Plan Decision #6 says "Lowercase + trim whitespace at insert time". The current implementation:
- Over-trims by stripping punctuation (e.g. an apostrophe in `it's` would be stripped from the lemma — wrong for German compound words too if there's any punctuation handling).
- Is **never called** — `upsertVocabularyEntry` takes a pre-formed `lemma:` parameter from the caller and never invokes `normalize`. The decision register said normalization happens "at insert time"; today, every caller is responsible for it.

Fix: either (a) call `normalize(lemma)` inside `upsertVocabularyEntry` so callers can pass surface forms directly, (b) trim only `.whitespaces` per Decision #6, and (c) delete the function if downstream tasks won't use it after all. I'd recommend (a) + (b).

**I2 — `BackupFamiliarityProgress.swift` is comment-only — dead file, but added to two source-membership build phases. [Introduced]**

`Shared/Data/Backup/Models/BackupFamiliarityProgress.swift` contains only a header comment explaining why the file is empty. It produces no symbols. It is still added to the `Aidoku` source build phase (twice — `LC0001AB` and `LC0001AC`) and to the project group. This is harmless to runtime but is a code smell and bloats the project tree.

Fix: delete the file and remove its pbxproj entries. Move the embedding rationale into a doc-comment at the top of `BackupVocabularyEntry.swift` (where it already partly lives).

**I3 — Vocabulary restore swallows save errors silently; no `BackupError.vocabulary` case. [Introduced]**

`Shared/Data/Backup/BackupManager.swift:541-551`:
```swift
let vocabularyTask = Task {
    if let backupVocab = backup.vocabulary {
        await CoreDataManager.shared.container.performBackgroundTask { context in
            for item in backupVocab {
                _ = item.toObject(context: context)
            }
            try? context.save()       // <— swallow
        }
    }
}
```

Sibling tasks (`mangaTask`, `historyTask`, `chaptersTask`, etc., e.g. `:318-336`) wrap saves in `do/try/catch`, return a `Bool`, and `throw BackupError.X` so the user gets a "Vocabulary failed to restore" alert. The vocab task drops errors on the floor. The user sees a "succeeded" restore even if the save crashed.

Fix:
1. Add `case vocabulary` to the `BackupError` enum (`BackupManager.swift:247-255`) with a localized string (e.g. `NSLocalizedString("VOCABULARY", ...)`).
2. Convert `vocabularyTask` to the same `Bool` + `throw` pattern as `historyTask`.
3. Add `case .vocabulary: NSLocalizedString("VOCABULARY")` to `stringValue`.

**I4 — `FlashcardState` is in the `Cloud` CloudKit configuration despite Decision #10 declaring it "ephemeral / session-only". [Introduced]**

`Shared/Aidoku.xcdatamodeld/0.9.0.xcdatamodel/contents:157`:
```
<configuration name="Cloud" usedWithCloudKit="YES">
    ...
    <memberEntity name="FlashcardState"/>
</configuration>
```

Plan Decisions #10 and #12 are in tension:
- #10: "flashcard state excluded from backup; review state is session-ephemeral".
- #12: "entities participate in existing NSPersistentCloudKitContainer setup, no opt-out".

The implementation followed #12 mechanically. The practical effect: a user's `sessionCorrect`/`lastShownAt` round-robin counters sync across all of their devices, which can cause a flashcard session on iPhone to spuriously skip cards on iPad. If the entity is "session-ephemeral" as #10 claims, it should be in the `Local` configuration (alongside `Source`), not `Cloud`.

Fix: move `FlashcardState` from the `Cloud` configuration to the `Local` configuration. Update Decision #12 in the plan to note the carve-out.

**I5 — No CloudKit deduplication handler for new entities. [Introduced]**

`Shared/Managers/CoreData/CoreDataManager.swift:193-200` lists the entity names tracked by the CloudKit remote-change handler, and `:243-272` defines `deduplicate(objectId:context:)` for those six types. With CloudKit enabled, the `(language, lemma)` uniqueness constraint on `VocabularyEntry` is **not enforced cross-device** — CloudKit doesn't honor uniqueness constraints. Two devices creating `(de-DE, "buch")` independently will both sync, and the user will see a duplicate row.

Fix: add an `else if let object = object as? VocabularyEntryObject` branch to `deduplicate(objectId:context:)` using `NSPredicate(format: "language == %@ AND lemma == %@", object.language, object.lemma)`. Add `VocabularyEntryObject.entity().name` to the `entityNames` array in `storeRemoteChange()`.

**I6 — `makeTestManager()` is unused dead code; the singleton-with-passed-context strategy it documents is what every test actually does, so the helper is just confusing. [Introduced]**

`AidokuTests/VocabularyManagerTests.swift:18-34` defines `makeTestManager()` but no test calls it. Every test uses `makeInMemoryContainer()` + `CoreDataManager.shared` + an explicitly-passed `ctx`. The dead helper's comment also includes a misleading hint ("Swap the singleton's container for the duration of this test by using…") that does not match what tests do.
