## Summary

The Learner stack is substantially complete and well-shaped. The data model, translation service stack with caching/composite/Foundation-Models routing, OCR service with NSCache-backed result cache, overlay coordinator, word lookup sheet, sentence translation sheet, vocabulary list, and flashcards view-model are all present and wired together via `LearnerEvents`. The earlier Task 2 build-blockers (C1/C2/C3 from `review-20260511-0716.md`) are resolved.

However, four AC-level gaps remain — three that break user-visible behavior on common code paths, and one that violates Task 8's explicit visibility-gating requirement. Most are scoped to single files and tightly bounded:

- **C1**: the Vocabulary tab is shown unconditionally to every user, ignoring Task 8 AC #2's `Learner.enabledGlobally` requirement (the key is never read anywhere in the codebase).
- **C2**: the OCR overlay only attaches on two of the four image-loading paths in `ReaderPageView`. Local-file (CBZ/ZIP) and base64 image sources — a primary Aidoku use case — silently get no overlay even when Learner mode is on.
- **C3**: word-tap inside the sentence sheet is unimplemented; sentences render as plain `Text` with no per-word interaction. Task 7 AC #4 was explicit about this.
- **C4**: the LLM grouping-output validation that Task 7's Risks section explicitly mandated as a fallback is not implemented — the view model only falls back when the LLM returns an empty array, not when it drops or duplicates fragments.

Additionally, two correctness issues that produce silent UX wrongness:

- **I9**: `LearnerOverlayView.hitTest` returns nil for empty-space hits, which routes those touches around the view's own `UILongPressGestureRecognizer`. The long-press sentence-translate trigger is effectively dead.
- **I11**: two different "normalize" functions with different rules (`LearnerStrings.normalizeLemma` strips punctuation; `VocabularyEntryObject.normalize` preserves it). Tapping a word with trailing punctuation stores it as `"buch"` but the overlay's badge lookup queries `"buch."` — the badge never appears.

Test quality is mixed. Several test suites suffer from the singleton-with-test-context impedance mismatch: tests seed entries in an in-memory test container but the view-model code under test uses `CoreDataManager.shared` with no context override, so the test's seeded data is invisible to the assertion path. The tests acknowledge this in comments and weaken assertions to tautologies ("isLoading == false"). I5 (testable composite re-implementation) and I4 (spy never wired) are similar smells.

Lint/typecheck/build verification is **INCONCLUSIVE**: this environment is on Command Line Tools (`xcode-select -p` → `/Library/Developer/CommandLineTools`), so `xcodebuild` cannot run. All correctness findings below are from static analysis of the diff, the existing convention, and read-through of the planned ACs. Recommended verification commands are in the Conclusion.

**C4 — Task 7 Risks mandated fragment-index validation; not implemented. Silent data loss when LLM grouping drops or duplicates fragments. [Introduced]**

Plan Task 7 Risks (verbatim): "the view model validates that every input fragment index appears exactly once across groups; if validation fails, fall back to a degenerate 'one fragment = one sentence' grouping (each line is its own sentence) and log the failure for tuning."

`iOS/UI/Learner/SentenceTranslationViewModel.swift:78-99`:
```swift
let groups: [SentenceGroup]
do {
    groups = try await service.groupFragmentsIntoSentences(fragments, language: sourceLanguage)
} catch let err as TranslationError {
    loadError = err
    ...
}

// Validate: if grouping returned nothing or appears invalid, fall back to one-line-per-sentence
let resolvedGroups: [SentenceGroup]
if groups.isEmpty {
    resolvedGroups = fragments.map { frag in
        SentenceGroup(fragmentIndices: [frag.index], combinedText: frag.text)
    }
} else {
    resolvedGroups = groups
}
```

The fallback is triggered only when `groups.isEmpty`. There is no check that `Set(groups.flatMap(\.fragmentIndices)) == Set(fragments.map(\.index))`.

Trigger trace (concrete):
1. Page has 4 OCR'd fragments at indices [0, 1, 2, 3].
2. Foundation Models returns 1 group: `SentenceGroup(fragmentIndices: [0, 1], combinedText: "Wort A Wort B")` — drops fragments 2 and 3 (LLM hallucination / partial output).
3. `groups.isEmpty` is false → fallback NOT triggered.
4. User sees 1 sentence in the sheet; fragments 2 and 3 are silently lost.

A second failure mode (duplicate indices) is also unguarded: an LLM may return `[SentenceGroup(fragmentIndices: [0, 1])]` and `[SentenceGroup(fragmentIndices: [1, 2])]` — the existing code accepts both.

Provenance: file is new in commit `13d991ab`.

Fix (smallest hunk):
```swift
let inputSet = Set(fragments.map(\.index))
let returnedFlat = groups.flatMap(\.fragmentIndices)
let returnedSet = Set(returnedFlat)
let isValid = !groups.isEmpty && returnedSet == inputSet && returnedFlat.count == inputSet.count
let resolvedGroups: [SentenceGroup] = isValid ? groups : fragments.map { frag in
    SentenceGroup(fragmentIndices: [frag.index], combinedText: frag.text)
}
if !isValid {
    print("[Learner] grouping returned invalid fragment indices; falling back to one-per-sentence")
}
```

The existing `TranslationServiceTests.grouping_fragmentIndices_coverInput` test asserts the invariant on a stub return value — but the view-model code never enforces it on real responses.

**I2 — `OCRServiceTests::recognize_offMainThread` doesn't verify what it claims. [Introduced]**

`AidokuTests/OCRServiceTests.swift:74-95`:
```swift
@Test func recognize_offMainThread() async throws {
    // ...
    var wasOnMainThread = true
    await Task.detached {
        do {
            _ = try await service.recognize(image: image, languages: ["de-DE"])
            wasOnMainThread = Thread.isMainThread   // ← checked AFTER recognize returns
        } catch {
            wasOnMainThread = false
        }
    }.value
    #expect(wasOnMainThread == false, ...)
}
```

The thread check happens AFTER `await service.recognize` returns. The continuation may resume on any thread the runtime chooses; what the test claims to verify (the Vision request callback runs off-main) is not actually tested. The `VisionOCRService.swift:35` assertion `assert(!Thread.isMainThread, "OCR must not run on main thread")` is what guarantees the property — the test should exercise that assertion.

Fix: the test as currently written can't really verify the property without invasive instrumentation. Either drop the test, or rewrite as: run `service.recognize(...)` from a `MainActor.run` block and verify the call completes without deadlocking (which would happen if OCR ran on main). Or expose an injection point on `VisionOCRService` for a "queue marker" the test can check.

**I3 — Tests reach into `CoreDataManager.shared` with in-memory test contexts, but view models call methods without context — assertions become tautologies. [Introduced]**

Examples:

`AidokuTests/VocabularyListViewModelTests.swift::refresh_loadsAllEntries`:
```swift
// Seed 3 entries via test ctx
_ = CoreDataManager.shared.upsertVocabularyEntry(..., context: ctx)  // <- test ctx
// ...
let vm = VocabularyListViewModel()
await vm.refresh()
// CoreDataManager.shared uses the app's viewContext, not our test context.
// We can only verify the call completes without error and isLoading resets.
#expect(vm.isLoading == false)   // <- tautology
```

`VocabularyListViewModel.refresh()` (line 65) calls `CoreDataManager.shared.getAllVocabulary()` with no `context:` argument — defaulting to `self.context`, i.e. the live persistent store. The seeded data is in the in-memory test container, invisible to refresh.

`AidokuTests/FlashcardsViewModelTests.swift::loadQueue_loadsEntries` has the same pattern.

This produces tests that pass on any implementation that doesn't crash. They don't verify the contract.

Fix: introduce a dependency-injection seam in the view models — e.g.
```swift
init(coreData: CoreDataManager = .shared, context: NSManagedObjectContext? = nil)
```
Then tests can pass the test container's context and assertions become meaningful. Compare `WordLookupViewModel`, which (mostly) accepts an explicit context through the entry's `objectID`.

**I10 — `WordLookupViewModel.lemma` normalization (strips punctuation) diverges from `VocabularyEntryObject.normalize` (preserves punctuation). Badges silently miss. [Introduced]**

```swift
}
```

`Shared/Data/Database/Objects/VocabularyEntryObject.swift:36-38`:
```swift

    lemma.lowercased().trimmingCharacters(in: .whitespacesAndNewlines)
}
```

Two different rules:
- `LearnerStrings.normalizeLemma("Buch.")` → `"buch"` (period stripped)
- `VocabularyEntryObject.normalize("Buch.")` → `"buch."` (period preserved)

Used at:
- `WordLookupViewModel.init(event:)` line 36: `self.lemma = LearnerStrings.normalizeLemma(event.surfaceForm)` → stores "buch"
- `LearnerOverlayView.swift:104`: `let lemma = VocabularyEntryObject.normalize(box.text)` → looks up "buch."
- `CoreDataManager+Vocabulary.upsertVocabularyEntry:80`: `let normalizedLemma = VocabularyEntryObject.normalize(lemma)` — note the *parameter* `lemma` here is the already-normalized-by-`LearnerStrings` version, so this is a no-op for word-tap inserts.

Trigger trace:
1. OCR detects "Buch." (with period) on a German page.
2. User taps the word; `WordTapEvent.surfaceForm = "Buch."`.
3. `WordLookupViewModel.init` computes `lemma = "buch"` (period stripped).
4. User taps "Add to vocab"; entry stored with `lemma = "buch"`.
5. Vocab changed event fires; `VocabIndex` rebuilds.
6. On the same (or next) page, `LearnerOverlayView.rebuild()` for the OCR word "Buch." computes `lemma = VocabularyEntryObject.normalize("Buch.")` → `"buch."`.
7. Lookup `vocabIndex[Identifier(language: "de-DE", lemma: "buch.")]` → nil (stored under "buch").
8. **No badge shown** even though the word is in vocab.

This violates PRD AC #4: "Words added to the vocab list show a familiarity badge over their bounding box on subsequent pages."

Conversely, a punctuation-bearing recognized form will never collide with a clean-form entry — see `VocabularyManagerTests::upsert_normalizesLemma`'s assertion that this is the *desired* property at storage time. But it's wrong at lookup time because the two layers disagree.

The test `upsert_normalizesLemma` at line 159-183 explicitly asserts the current divergent behavior — it would need to be updated.

Recommend: change `LearnerStrings.normalizeLemma` to match `VocabularyEntryObject.normalize` (drop the `.punctuationCharacters` trim). Then the lock-in test `upsert_normalizesLemma` already encodes the right invariant; only `normalizeLemma_stripsPunctuationAndLowercases` in `WordLookupViewModelTests.swift:44-49` needs to flip its assertions.

**I12 — `ReaderViewController.viewWillDisappear` cancels Combine subscriptions; no re-subscription path. If the reader VC is re-shown without a fresh instance, word/sentence events stop reaching it. [Introduced]**

`iOS/UI/Reader/ReaderViewController.swift:345-348`:
```swift

    super.viewWillDisappear(animated)
    wordTapSubscription?.cancel()
    sentenceTranslateSubscription?.cancel()
    // ...
```

But subscriptions are set up in `observe()` (called once from base class lifecycle), not in `viewWillAppear`. After cancellation in `viewWillDisappear`, there is no re-subscription path.

Mitigating factor: in current Aidoku usage, the reader is presented modally with a new instance each time, so `viewDidLoad` → `observe()` runs again and the subscriptions reconnect. So in practice this isn't a visible bug.

Fix: move cancellation to `deinit`, or re-subscribe in `viewWillAppear`.

This is preexisting in the patterns used elsewhere in `ReaderViewController` — `addObserver(forName:)` calls in `observe()` are also one-shot — so it's consistent with the existing convention. Acceptable, with a comment noting the assumption.
