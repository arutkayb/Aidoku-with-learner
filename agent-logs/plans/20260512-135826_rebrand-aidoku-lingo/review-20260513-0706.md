## Findings (Introduced / Aggravated)

### Critical

#### Critical #1: URL-scheme rename silently breaks every preexisting `aidoku://` consumer  `[Aggravated]`

**Location**: `iOS/Info.plist:61` (the change) vs. consumers below.

**Triggering scenarios**:
- User taps "Login with Shikimori" → OAuth provider redirects to `aidoku://shikimori-auth?code=...` → iOS has no handler registered for `aidoku://` (only `aidoku-lingo://`) → callback fails, login broken.
- Same flow for Bangumi, MangaBaka, OIDC self-hosted source setup.
- User opens an `aidoku://addSourceList?url=...` deep link from Safari → iOS no longer recognizes the scheme; nothing happens.
- Even if a `aidoku-lingo://` URL is opened, `iOS/AppDelegate.swift:467` checks `url.scheme == "aidoku"` — that branch is now dead code.

**Trace**:
1. `iOS/Info.plist:61` now declares only `aidoku-lingo` in `CFBundleURLSchemes` (commit `60384f659`, blame confirmed).
2. iOS launches `LSApplicationQueriesSchemes` / URL routing using exactly what `CFBundleURLSchemes` registers. `aidoku://` is no longer routed to this bundle.
3. 10 in-code call-sites still produce or expect `aidoku://`:
   - `iOS/AppDelegate.swift:467` — `if url.scheme == "aidoku"`
   - `Shared/Tracking/Trackers/shikimori/ShikimoriApi.swift:38`, `ShikimoriTracker.swift:56`
   - `Shared/Tracking/Trackers/bangumi/BangumiApi.swift:42,65`, `BangumiTracker.swift:131`
   - `Shared/Tracking/Trackers/mangabaka/MangaBakaApi.swift:34`, `MangaBakaTracker.swift:111,120`
   - `iOS/New/Views/Browse/SelfHostedSourceSetupView.swift:561`
4. Tracker OAuth providers (Shikimori/Bangumi/MangaBaka) have these redirect URIs pre-registered on the provider side. Even if we patch the strings locally, the providers will still redirect to `aidoku://…`.
5. Build verifies happy path only — runtime failure is silent until login is attempted.

**Recommended fix** (lowest risk): keep both schemes in `CFBundleURLSchemes`:
```xml
<array>
    <string>aidoku-lingo</string>
    <string>aidoku</string>
</array>
```
This preserves the upstream OAuth-callback contract (which external providers cannot easily be reconfigured for) and lets `aidoku-lingo://` coexist for future use. `AppDelegate.handleUrl` already checks `url.scheme == "aidoku"`, so it keeps working unchanged.

**Evidence commands**:
- `git blame iOS/Info.plist -L 55,70` → line 61 changed by commit `60384f659` (Task 1).
- `rg "aidoku://" --type swift | wc -l` → 10 matches.
- `git blame iOS/AppDelegate.swift -L 467,467` → preexisting (Skitty 2022).
- `git blame .../ShikimoriApi.swift -L 38,38` → preexisting (Vladimir 2025).
